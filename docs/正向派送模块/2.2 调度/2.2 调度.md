在上一章中已说明货物在仓库进行分拣,主要服务于两个目的:
1. 录入当日批次
2. 根据订单的 FSA 属性判断其对应的路线号,使其落入正确格口,送往对应的二级仓

以上两点均是调度的**前置条件**——在系统层面,批次是批量调度的执行单位;在物理层面,包裹与仓库之间建立了联系。而调度要做的事,就是把"包裹—二级仓/路线"的对应关系,进一步细化为"包裹—司机"的对应关系。

### 调度流程 -- 管理端
正向派送业务的调度,在管理端涉及的模块位置如下:

管理端
├─ 调度管理
│   └─ 路径规划
│       ├─ 派送规划
│       └─ 快速转单
└─ 订单管理
    └─ 中大件订单列表

| 模块      | 路径                 | 功能                        |
| ------- | ------------------ | ------------------------- |
| 派送规划    | 调度管理 → 路径规划 → 派送规划 | 批量分配当日批次的订单到对应路线号所属司机     |
| 快速转单    | 调度管理 → 路径规划 → 快速转单 | 处理特殊情况造成的需要给订单更换司机        |
| 中大件订单列表 | 订单管理 → 中大件订单列表     | 可查验司机分配情况,及后续预扫描节点触发的轨迹节点 |

### 调度流程 -- 库内
在调度阶段,库内的主要操作发生在**分配司机后、司机取货前**这个时间点:

1. 仓库人员用 App 对包裹进行**预扫描**
2. 根据已分配的司机,在包裹上写上该包裹在司机当日任务中的**排单号**
3. 通过这一步,将"司机—包裹"的关联性,从管理端系统层面扩展到实体包裹层面

预扫描完成后,数据会同步映射回管理端——在**中大件订单列表**中,该包裹会新增一个 **"已代扫"** 节点。
